# Architecture Notes

## Professional Mode Rendering

Professional Mode is the default mode. It uses the same app routes but filters toward recruiter-facing content:

- `app/page.tsx` renders a SaaS-inspired hero, professional headshot, metrics, selected experience, and core skills.
- `app/experience/page.tsx` shows only `professionalExperiences`.
- `app/projects/page.tsx` shows only projects where `mode === "professional"`.
- Detail pages are generated statically from the shared data arrays.

The visual language is intentionally minimal: soft gradients, glass cards, rounded geometry, and high-contrast typography inspired by Linear, Vercel, Notion, and Stripe.

## Creative Mode Rendering

Creative Mode is not a theme swap. It changes the visible content and interaction model:

- Corporate / traditional experience is hidden.
- `creativeExperiences` replaces the professional timeline.
- Creative projects are shown as “builds.”
- The homepage becomes a Minecraft-inspired overworld with pixel borders, terrain colors, and an interactive avatar panel.

The mode is stored in `localStorage` under `portfolio-mode` and mirrored onto `document.documentElement.dataset.mode`.

## Experience Categorization Logic

The LinkedIn PDF was categorized by content intent:

- **Professional**: TPM work, operations, SDLC, enterprise systems, AI platforms, analytics, procurement, research operations, executive reporting, and measurable delivery outcomes.
- **Creative**: student media, storytelling, music-club content, entrepreneurship curation, aesthetics, community initiatives, visual direction, and creative leadership.

If an item could fit both, the chosen mode is based on how it should feel to a visitor:

- Recruiter-facing execution work goes Professional.
- Identity, storytelling, community, and visual work goes Creative.

Creative Mode intentionally excludes corporate resume-style roles.

## Project Page Generation

Projects live in `data/portfolio.ts`.

Each project supports:

- `overview`
- `problem`
- `solution`
- `impact`
- `technologies`
- `screenshots`
- `links`

Routes are generated by `app/projects/[slug]/page.tsx` via `generateStaticParams()`, which makes them compatible with static export and GitHub Pages.

## 3D Minecraft Avatar Implementation

The current implementation is a lightweight pseudo-3D avatar in `components/Avatar3D.tsx`.

Why this path:

- GitHub Pages static export stays simple.
- No WebGL dependency is required.
- It remains responsive and fast-loading.
- It supports mouse / pointer drag rotation.

The component uses CSS perspective and transform updates:

- Pointer drag changes X/Y rotation.
- The Minecraft avatar image floats inside a stylized card.
- The result feels like a rotatable Minecraft character presentation without requiring Three.js.

Future upgrade path:

1. Add `three`, `@react-three/fiber`, and `@react-three/drei`.
2. Convert or source a true Minecraft skin texture.
3. Render cuboid body parts with UV-mapped skin textures.
4. Keep `Avatar3D.tsx` as the integration boundary.

## Component Architecture

- `ModeProvider`: client state, persistence, and mode toggle behavior.
- `ModeToggle`: accessible animated switch.
- `Header`: shared navigation and mode control.
- `Shell`: mode-aware visual surface wrapper.
- `Card`: reusable glass card for experience and project summaries.
- `SectionHeader`: consistent page heading pattern.
- `Avatar3D`: Creative Mode interactive avatar presentation.
